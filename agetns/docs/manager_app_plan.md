# План розробки екосистеми мобільних додатків Rekognita

## 1. Загальна архітектура системи (RBAC)
Як ви і пропонували, найбезпечнішим та найбільш логічним рішенням буде розділення функціоналу на **два окремі мобільні додатки** (або дві повністю ізольовані зони в одному, але для кращої сек'юрності та простоти юзкейсу краще два):

1. **Rekognita Manager** — додаток для керівників та адміністраторів компанії-клієнта.
2. **Rekognita Scanner (Worker App)** — максимально простий додаток для кінцевих співробітників (тих, хто робить фото).

Цей документ фокусується на плані розробки **Rekognita Manager**, який ми будемо розробляти першим у цьому репозиторії.

## 2. Додаток 1: Rekognita Manager (Фокус поточної розробки)

### Головна мета
Дати можливість компаніям самостійно та гнучко налаштовувати процес оцифрування документів, керувати співробітниками та інтеграціями з їхніми внутрішніми системами.

### 2.1. Основні модулі (Features)

#### Модуль 1: Управління Командою (Team/RBAC)
- **Запрошення співробітників:** Менеджер може генерувати доступи для працівників на місцях.
- **Права доступу:** Налаштування того, які типи документів може бачити та завантажувати конкретний працівник або відділ.
- **Блокування доступу:** Можливість в 1 клік відкликати доступ у звільненого співробітника.

#### Модуль 2: Управління Типами документів (Document Types)
- **Створення категорій:** Наприклад, "Накладні", "Медичні довідки", "Договори", "Паспорти".
- **Налаштування типу:** Налаштування обов'язковості багатосторінкового фото, додаткові описи для працівників.

#### Модуль 3: Інтеграції (Routing & Integrations)
- **Мапінг систем:** Прив'язка типів документів до конкретних систем.
   - *Приклад 1:* Всі "Накладні" автоматично летять через API в **1С/БАС** бухгалтерії.
   - *Приклад 2:* "Медичні довідки" відправляються в спеціалізовану медичну CRM або на Email HR-менеджеру.
- **Підключення:** Простий UI для вдалих налаштувань Webhook URL ваших клієнтів.

#### Модуль 4: Шаблони розпаршування (Parsing Templates)
- **Налаштування екстракції:** Оскільки система заснована на мультимодальних OCR моделях, менеджер може створити конфігурацію того, **що саме** треба дістати з фото.
- **Поля для пошуку:** 
   - Наприклад, вказати: "Знайди таблицю з товарами, дістань артикул, ціну, кількість", "Знайди ПІБ", "Знайди дату документу".
   - Визначення структури JSON на виході для подальшої відправки в інтеграції.
- **Додаткові інструкції (Prompts):** Тонке налаштування для AI-моделі під специфіку документів конкретної компанії.

## 3. Додаток 2: Rekognita Scanner (Worker App) - Короткий огляд
- **Юзкейс:** Співробітник логіниться -> Бачить лише велику кнопку "Сфотографувати" -> Вибирає тип з доступних йому (напр., "Накладна") -> Робить фото -> Перевіряє якість -> Відправляє.
- **Сек'юрність (Maximum Security):**
  - Фотографії робляться напряму через камеру в оперативну пам'ять (RAM) і **не зберігаються в галереї смартфона**.
  - Відправка відбувається по захищеному каналу.
  - Пристрій не зберігає кеш розпізнаних даних або історію після успішної відправки.
  - Повністю ізольовано від публічних LLM (ніякого відкритого ChatGPT API), використовується закритий контур безпеки.

## 4. Дизайн та UI/UX (SaaS-вигляд)
Для **Manager App** застосовується брендинг Rekognita:
- **Кольори:** Акцентний яскраво-синій `#3b82f6` та насичений `#2563eb` на кнопках і важливих діях. Глибокий графітовий `#0f172a` для заголовків та читабельного тексту. Фони `#f1f5f9` (світло-сіро-блакитний) і білий `#ffffff` для карток, що створює "чистий" SaaS-вигляд.
- **Стиль:** Сучасний, мінімалістичний, з використанням Card-дизайну для структур даних.
- **Планшетний режим:** Обов'язкова адаптація UI під планшети (бо менеджерам зручніше робити складні налаштування на великих екранах).

## 5. План розробки (Roadmap) для Manager App (Flutter)

Оскільки в папці проекту вже є структура Flutter-додатку (папки `lib`, `android`, `ios`), ми продовжимо його як базу для Rekognita Manager.

### Фаза 1: Ініціалізація та Базова Архітектура (Project Setup)
- [ ] Організація структури папок у `lib/` (використання Feature-first або Clean Architecture).
- [ ] Налаштування теми (`ThemeData`), підключення фірмових кольорів та шрифтів (створення `theme.dart`).
- [ ] Налаштування State Management та Routing (наприклад, GoRouter для зручної навігації).

### Фаза 2: Авторизація та Сек'юрне сховище (Auth & Security)
- [ ] Екрани входу в систему (Login).
- [ ] Збереження Auth-токенів у безпечному сховищі (Flutter Secure Storage).
- [ ] Головне меню / Дашборд менеджера (скелет екрану).

### Фаза 3: Керування Співробітниками (Team Management)
- [ ] Створення UI для списку співробітників та їх статусів.
- [ ] Екран додавання нового співробітника та налаштування його ролей/доступів до певних Типів Документів.

### Фаза 4: Управління Типами документів (Document Types)
- [ ] Список типів документів.
- [ ] Форма створення/редагування нового Типу документа з його властивостями.

### Фаза 5: Шаблони розпаршування (Template Builder) та Інтеграції
- [ ] Інтерфейс додавання полів для екстракції даних (Назва поля, Тип: текст, дата, число, таблиця).
- [ ] UI для налаштування куди відправляти (Webhook Builder) для конкретного типу документу.
- [ ] Зв'язування шаблону, інструкцій (prompts) та інтеграції в єдиний флоу.

### Фаза 6: Зв'язок з API та Полірування
- [ ] Підключення розробленого UI до реального API бекенду.
- [ ] Обробка станів завантаження, помилок (Error Handling), Toast-повідомлення.
- [ ] Тестування та збірка тестової версії.
